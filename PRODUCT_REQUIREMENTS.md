# Product Requirements

## 1. 背景

求职者通常拥有一份通用简历，但不同岗位 JD 对技能、项目经验、业务理解和表达重点的要求不同。用户手动为每个岗位改简历成本高，也容易出现两个问题：

- 重点不匹配：简历没有突出岗位最关心的能力。
- 表达不可信：为了匹配 JD 过度包装，导致面试时难以自洽。

ResumeFit 的目标不是替用户编造经历，而是帮助用户基于真实经历进行结构化梳理、岗位化表达和风险提醒。

## 2. 产品目标

### Demo 阶段目标

Demo 阶段要证明以下闭环成立：

1. 用户可以录入通用简历和项目经历。
2. 用户可以粘贴目标岗位 JD。
3. 系统可以分析 JD 的核心要求。
4. 系统可以评估简历与 JD 的匹配度。
5. 系统可以生成一版基于真实信息的岗位定制 Markdown 简历。
6. 系统可以提示真实性风险和可能的面试追问。
7. 用户可以保存多个岗位版本。

### 商业化阶段目标

商业化阶段可继续扩展：

- 账号体系和云端同步。
- PDF/Word 导出。
- 会员套餐和支付。
- 多模型供应商配置。
- 简历模板市场。
- 求职进度管理。
- 面试准备工作台。
- 数据分析和个性化推荐。

## 3. 目标用户

### MVP 主要用户

- 准备投递多个岗位的应届生、转岗者或社招候选人。
- 有真实项目经历，但不知道如何针对 JD 调整表达的人。
- 希望提前知道简历风险和面试追问点的人。

### 暂不服务的用户

- 企业 HR 和招聘团队。
- 猎头机构。
- 需要批量处理候选人简历的 B 端用户。
- 希望自动爬取招聘网站并批量投递的用户。

## 4. 核心用户流程

1. 用户创建或编辑通用简历。
2. 用户维护项目库，包括项目背景、职责、技术栈、成果和可证明材料。
3. 用户粘贴岗位 JD。
4. 系统分析 JD，提取岗位职责、硬技能、软技能、加分项和隐含要求。
5. 系统对简历和 JD 进行匹配评分。
6. 系统输出缺口、优势、风险和优化建议。
7. 用户选择生成岗位定制简历。
8. 系统生成 Markdown 简历版本。
9. 系统给出真实性风险检测结果。
10. 系统生成可能的面试追问。
11. 用户保存、查看和导出不同岗位版本。

## 5. 功能需求

### 5.1 通用简历输入

用户可以输入或编辑：

- 基本信息
- 求职方向
- 教育经历
- 工作经历
- 技能清单
- 项目经历摘要
- 自我评价
- 其他补充材料

MVP 可先使用结构化表单加 Markdown 文本区，后续再支持简历文件解析。

### 5.2 项目库管理

项目库用于承载用户真实经历，是简历定制的事实来源之一。

项目字段包括：

- 项目名称
- 项目时间
- 项目背景
- 用户角色
- 技术栈
- 关键职责
- 量化成果
- 难点与解决方案
- 可被面试追问的细节
- 真实性备注

### 5.3 岗位 JD 分析

系统应从 JD 中提取：

- 岗位名称
- 公司或团队背景
- 核心职责
- 必备技能
- 加分技能
- 业务领域
- 经验年限要求
- 关键词
- 隐含能力要求
- 简历应突出方向

### 5.4 简历匹配度评分

系统应生成匹配报告，包括：

- 总分
- 技能匹配
- 项目匹配
- 行业或业务背景匹配
- 经验年限匹配
- 表达匹配
- 主要优势
- 主要短板
- 优化建议

评分用于辅助判断，不应伪装成招聘方真实筛选结果。

### 5.5 岗位定制简历生成

系统基于通用简历、项目库、JD 分析和匹配报告，生成一版岗位定制 Markdown 简历。

生成原则：

- 只能重组、压缩、强调和改写已有真实信息。
- 不允许新增不存在的公司、项目、岗位、学历、证书、指标或技术经验。
- 对不确定信息应保持保守表达。
- 优先突出与 JD 相关的项目和技能。

### 5.6 真实性风险检测

系统检测简历版本中的风险：

- 与原始资料不一致。
- 表述过度绝对化。
- 技术深度可能无法支撑。
- 量化指标缺少来源。
- 项目职责和职位级别不匹配。
- JD 关键词堆砌痕迹明显。

输出应包括风险等级、风险说明和修改建议。

### 5.7 面试追问预测

系统基于 JD 和定制简历预测面试官可能追问：

- 项目细节问题
- 技术原理问题
- 架构设计问题
- 业务理解问题
- 指标真实性问题
- 冲突和取舍问题

每个问题可附带回答准备建议，但 MVP 不要求生成完整答案。

### 5.8 多版本简历管理

用户可以保存多个 `ResumeVersion`：

- 原始通用版本
- 针对不同 JD 的定制版本
- 手动修改后的版本

每个版本应关联目标 JD、匹配报告和生成来源。

### 5.9 Markdown 导出

MVP 支持 Markdown 导出，便于用户复制、提交或后续转成 PDF/Word。

后续可扩展：

- PDF 导出
- Word 导出
- 多模板样式
- 中英文版本

## 6. 非功能需求

- 简洁：Demo 页面应优先服务核心流程，不做复杂后台。
- 可解释：AI 结果应能展示依据和风险。
- 可替换：AI 服务商通过配置切换。
- 可维护：业务逻辑分模块，不堆在单一大文件。
- 可迁移：SQLite 模型设计预留 PostgreSQL 升级空间。
- 安全：API Key 只能通过环境变量或安全配置读取。

## 7. MVP 明确不做

- 支付、会员和订阅。
- 招聘网站爬取或自动投递。
- 手机 APP。
- 企业后台。
- 多人协作。
- 复杂权限系统。
- 简历文件 OCR。
- PDF/Word 高保真排版。
